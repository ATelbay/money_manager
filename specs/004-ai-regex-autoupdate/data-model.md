# Data Model: AI-Driven Parser Auto-Update

**Date**: 2026-03-11 | **Branch**: `004-ai-regex-autoupdate`

## Entities

### ParserConfig (existing — extended)

No schema changes to the existing `ParserConfig` data class. The AI-generated configs use the exact same model with all existing fields.

**Fields (unchanged)**:
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| bankId | String | yes | Lowercase latin slug (e.g., "kaspi", "halyk_bank") |
| bankMarkers | List\<String\> | yes | Substrings to detect bank in PDF text |
| transactionPattern | String | yes | Regex pattern for transaction rows |
| dateFormat | String | yes | Date format pattern (e.g., "dd.MM.yyyy") |
| operationTypeMap | Map\<String, String\> | yes | Operation name → "income"\|"expense" |
| skipPatterns | List\<String\> | no | Lines to skip (header/footer filters) |
| joinLines | Boolean | no | Join continuation lines |
| amountFormat | String | no | Amount parsing format ("space_comma", "comma_dot", "dot") |
| useSignForType | Boolean | no | Use +/- sign to determine transaction type |
| negativeSignMeansExpense | Boolean | no | Negative sign = expense |
| useNamedGroups | Boolean | no | Use named regex groups (date, sign, amount, operation, details) |
| deduplicateMaxAmount | Boolean | no | Keep max amount per date+details group |

### ParserCandidate (new — Firestore only)

Represents a crowd-validated parser configuration awaiting promotion. Stored in Firestore `parser_candidates` collection (global, not per-user).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | String | auto | Firestore document ID (auto-generated) |
| bankId | String | yes | Normalized bank slug from AI |
| transactionPattern | String | yes | The core regex pattern (used for dedup matching) |
| parserConfigJson | String | yes | Full serialized ParserConfig JSON |
| anonymizedSample | String | yes | Sanitized table rows (merchants replaced, accounts removed) |
| userIdHash | String | yes | SHA-256 hash of the submitting user's ID |
| successCount | Int | yes | Number of independent successful imports (starts at 1) |
| status | String | yes | "candidate" \| "promoted" \| "rejected" |
| createdAt | Long | yes | Epoch millis of first submission |
| updatedAt | Long | yes | Epoch millis of last successCount update |

**Identity & Uniqueness**: Candidates are matched by `bankId + transactionPattern`. Before creating a new document, the client queries for existing candidates with the same pair and increments `successCount` if found.

**State Transitions**:
```
candidate → promoted    (when successCount >= threshold, via Cloud Function)
candidate → rejected    (manual, if config conflicts with existing bank)
promoted  → (terminal)  (no further state changes)
```

### CachedAiConfig (new — DataStore)

Local cache of AI-generated configs stored in Preferences DataStore as a single JSON string.

| Key | Type | Description |
|-----|------|-------------|
| ai_parser_configs | String (JSON) | Serialized list of ParserConfig objects generated by AI |

**Lifecycle**:
- Written after a successful AI-generated config is used to parse transactions
- Read before invoking Gemini (check if a cached config exists for the detected bank)
- Becomes redundant when the config is promoted and delivered via Remote Config
- No automatic cleanup — entries accumulate but remain small (few KB)

### AnonymizedSample (value object — not persisted independently)

A sanitized representation of statement table rows, embedded in ParserCandidate.

**Anonymization Rules**:
1. All merchant/recipient names → "MERCHANT_1", "MERCHANT_2", ... (sequential per unique name)
2. All account/card numbers (sequences of 4+ digits possibly with `****` masks) → removed
3. Amounts → preserved (needed for regex validation)
4. Dates → preserved (needed for dateFormat validation)
5. Currency codes → preserved
6. Column structure/whitespace → preserved

## Relationships

```
Remote Config (parser_configs)
    ├── [existing 5 bank configs]
    └── [promoted configs from Cloud Function]

DataStore (ai_parser_configs)
    └── [locally cached AI-generated configs]

Firestore (parser_candidates)
    └── [candidate configs from all users]
         ↓ (Cloud Function on successCount >= threshold)
    Remote Config (parser_configs) — appended

ParseStatementUseCase resolution order:
    1. Remote Config configs (official + promoted)
    2. DataStore cached AI configs
    3. Generate new config via Gemini
    4. Full AI fallback (existing behavior)
```
