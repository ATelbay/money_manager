# Implementation Plan: AI-Driven Parser Auto-Update

**Branch**: `004-ai-regex-autoupdate` | **Date**: 2026-03-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-ai-regex-autoupdate/spec.md`

## Summary

When a user imports a PDF from an unsupported bank, the system extracts a sample of table rows, sends them to Gemini AI to generate a `ParserConfig`-compatible regex configuration (via structured output), validates it against ReDoS patterns, and parses the full PDF locally with the existing regex engine. Successfully imported configs are anonymized and submitted to a shared Firestore candidate pool. When a candidate reaches 3 independent successful imports, a Cloud Function auto-promotes it to Remote Config so all users get the new bank without an app update.

**Key architectural decision**: No new Gradle modules — extend existing `core:parser`, `core:ai`, `core:firestore`, `core:datastore`, `domain:import`, and `presentation:import`. The only new project is a Cloud Function in `functions/`.

## Technical Context

**Language/Version**: Kotlin 2.3.0 (Android), TypeScript (Cloud Function)
**Primary Dependencies**: Firebase AI SDK (Gemini 2.5 Flash), Firestore, Remote Config, PdfBox-Android 2.0.27.0, kotlinx-serialization 1.10.0
**Storage**: Room (existing transactions/categories), Firestore (parser_candidates collection), DataStore (cached AI configs), Remote Config (promoted configs)
**Testing**: JUnit 4 + MockK 1.14.9 + Turbine 1.2.1 (unit), ComposeTestRule (UI)
**Target Platform**: Android (minSdk per project config)
**Project Type**: Mobile app + Cloud Function
**Performance Goals**: AI config generation < 10s, regex parsing < 1s, import flow < 30s total
**Constraints**: Offline-capable for local parsing, online required for AI generation and candidate submission
**Scale/Scope**: ~50 active users initially, candidate pool grows to ~50-100 configs
