# Feature Specification: Animation Audit & Motion System

**Feature Branch**: `001-animation-audit`
**Created**: 2026-03-08
**Status**: Draft
**Input**: Провести аудит и улучшение всех анимаций — создать централизованную motion-систему, исправить баг смены темы, унифицировать все анимации

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Плавные и консистентные анимации во всём приложении (Priority: P1)

Пользователь взаимодействует с приложением и видит, что все анимации (навигационные переходы, нажатия кнопок, выбор элементов, появление данных) работают плавно, без рывков и задержек. Все переходы между экранами выглядят одинаково. Нет ощущения, что разные части приложения сделаны разными людьми.

**Why this priority**: Это основная цель аудита — привести все 13+ анимаций к единой системе токенов. Без этого остальные улучшения не имеют фундамента.

**Independent Test**: Можно протестировать, открывая каждый экран приложения и наблюдая за консистентностью переходов. Значение — приложение воспринимается как профессиональный, цельный продукт.

**Acceptance Scenarios**:

1. **Given** пользователь находится на любом экране, **When** он переходит на другой экран через навигацию, **Then** переход происходит с одинаковой анимацией (длительность, easing, тип) на всех экранах
2. **Given** пользователь нажимает на кнопку или chip, **When** элемент реагирует на нажатие, **Then** feedback-анимация использует единые параметры из motion-системы
3. **Given** пользователь открывает экран со списком элементов, **When** данные загрузились, **Then** элементы появляются плавно без рывков и пропуска кадров
4. **Given** приложение работает на устройстве среднего класса, **When** пользователь активно взаимодействует с интерфейсом, **Then** анимации не вызывают заметных подтормаживаний (dropped frames)

---

### User Story 2 — Корректная смена темы без артефактов (Priority: P2)

Пользователь заходит в Настройки и переключает тему (светлая/тёмная/системная). Приложение плавно переходит к новой цветовой схеме без мерцаний, вспышек неправильных цветов или показа контента другого экрана.

**Why this priority**: Это прямой баг, который портит впечатление. Пользователь видит на долю секунды неправильный экран — это выглядит сломанным.

**Independent Test**: Можно протестировать, переключая тему в настройках многократно. Значение — устраняет видимый баг, улучшает polish приложения.

**Acceptance Scenarios**:

1. **Given** пользователь находится на экране Настроек в светлой теме, **When** он выбирает тёмную тему, **Then** цвета меняются плавно без показа контента другого экрана и без вспышек
2. **Given** пользователь находится в тёмной теме, **When** он переключает на системную тему (которая сейчас светлая), **Then** переход происходит без артефактов
3. **Given** пользователь быстро переключает тему несколько раз подряд, **When** анимации ещё не завершились, **Then** приложение не крашится и корректно отображает финальную выбранную тему

---

### User Story 3 — Централизованная motion-система для разработчиков (Priority: P3)

Разработчик добавляет новый экран или компонент в приложение. Он использует готовые токены длительностей, easing-кривых и reusable composable из motion-системы вместо того, чтобы придумывать свои значения. Все новые анимации автоматически консистентны с остальным приложением.

**Why this priority**: Это инфраструктурная задача — создание переиспользуемой системы. Она обеспечивает долгосрочную консистентность, но сама по себе не меняет UX для конечного пользователя.

**Independent Test**: Можно протестировать, создав новый экран и проверив, что разработчик может использовать токены из motion-системы без хардкода. Значение — предотвращает повторное появление проблемы хаотичных анимаций.

**Acceptance Scenarios**:

1. **Given** разработчик создаёт новый компонент, **When** ему нужна анимация для навигационного перехода, **Then** он находит готовый набор duration/easing токенов в motion-системе
2. **Given** разработчик смотрит существующий код, **When** он ищет анимационные параметры, **Then** все компоненты ссылаются на централизованные токены, а не на хардкод числа
3. **Given** motion-система определена в одном месте, **When** дизайнер решает изменить общую скорость анимаций, **Then** достаточно поменять значение в одном файле, и все экраны обновятся

---

### Edge Cases

- Что происходит при прерывании анимации (пользователь нажимает «Назад» во время перехода)?
- Как ведут себя анимации при низкой производительности устройства / включённом режиме энергосбережения?
- Что происходит с анимациями при включённых Accessibility-настройках «Убрать анимации» (Reduce Motion)?
- Как ведёт себя staggered-анимация списков при большом количестве элементов (100+ категорий)?
- Что происходит с shared element transition при быстром двойном тапе на транзакцию?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение ДОЛЖНО иметь централизованный набор анимационных токенов (duration, easing) доступный из всех presentation-модулей
- **FR-002**: Все навигационные переходы между экранами ДОЛЖНЫ использовать одинаковые параметры анимации из motion-системы
- **FR-003**: Переходы между bottom navigation destinations ДОЛЖНЫ использовать crossfade, а drill-in переходы (детали транзакции, редактирование аккаунта) ДОЛЖНЫ использовать slide+fade (Material 3 стандарт)
- **FR-004**: Все feedback-анимации (нажатие кнопок, выбор chip, сегментов) ДОЛЖНЫ использовать единые spring/tween параметры из motion-системы
- **FR-005**: Смена темы ДОЛЖНА происходить без показа артефактов (вспышки неправильного экрана, мерцания)
- **FR-006**: Анимации числовых значений (баланс, суммы) ДОЛЖНЫ использовать единый duration-токен из motion-системы
- **FR-007**: Staggered-анимации списков (категории в статистике) и все chart-анимации (donut, bar) ДОЛЖНЫ использовать токены задержки и длительности из motion-системы
- **FR-008**: Приложение ДОЛЖНО уважать системную настройку Accessibility «Reduce Motion» — при включённой настройке длительность всех анимаций сокращается до минимума (~0-50ms), но тип перехода сохраняется для контекста навигации
- **FR-009**: Все хардкод значения длительностей и easing в существующих компонентах ДОЛЖНЫ быть заменены ссылками на motion-токены
- **FR-010**: Анимации НЕ ДОЛЖНЫ вызывать лишних recomposition (использовать graphicsLayer, lambda-based modifiers где возможно)

### Key Entities

- **Motion Tokens**: Набор именованных констант для duration (short, medium, long) и easing (standard, decelerate, accelerate) — единый источник правды для всех анимаций
- **Navigation Transitions**: Конфигурации enter/exit/popEnter/popExit для двух типов переходов: tab-switch и drill-in
- **Theme Transition**: Механизм плавного перехода между цветовыми схемами при смене темы

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% навигационных переходов используют токены из motion-системы (0 хардкод значений в navigation-конфигурации)
- **SC-002**: 100% компонентов с анимациями ссылаются на централизованные токены (0 хардкод duration/easing в presentation и core:ui модулях)
- **SC-003**: Смена темы происходит без видимых артефактов — нет показа неправильного экрана даже на долю секунды
- **SC-004**: Приложение корректно реагирует на системную настройку Reduce Motion
- **SC-005**: Все анимации завершаются за визуально комфортное время (не более 1 секунды для любой анимации)
- **SC-006**: При прокрутке списков и навигации между экранами не наблюдается заметных пропусков кадров на устройствах среднего класса

## Clarifications

### Session 2026-03-08

- Q: Как должны отличаться tab-switch и drill-in переходы? → A: Crossfade для tab-switch, slide+fade для drill-in (Material 3 стандарт)
- Q: Как вести себя при Reduce Motion? → A: Сократить длительность до минимума (~0-50ms), сохранив тип перехода
- Q: Токенизировать ли premium animation layers (charts, stagger)? → A: Да, всё — stagger delay и chart durations тоже становятся токенами

## Assumptions

- Material 3 предоставляет достаточно гибкости для кастомизации навигационных переходов
- Текущий баг смены темы вызван использованием двух независимых NavController в circular reveal: `bottomNavController` всегда начинает с `startDestination` (Home), поэтому при переключении темы на экране Settings пользователь видит Home на долю секунды. Исправление: один NavController + graphicsLayer snapshot (см. research.md R-005)
- Motion-система будет размещена в модуле `core:ui`, так как он уже содержит shared Compose компоненты и тему
- «Reduce Motion» реализуется через `AccessibilityManager` или `ComposeView` accessibility API
- Shared element transitions (ExperimentalSharedTransitionApi) остаются как есть — аудит фокусируется на стандартных анимациях
