Universal Self-Learning Bank Statement Parser
=============================================

Идея: Gemini генерирует regex-конфиг для неизвестного банка, а не парсит транзакции напрямую.
Один раз на банк → дальше regex работает точно, оффлайн, с дедупликацией.

## Флоу

1. PDF → извлечь текст → банк известен?
   - ДА → Regex (быстро, бесплатно, оффлайн)
   - НЕТ → Gemini анализирует СТРУКТУРУ выписки → генерирует ParserConfig
2. Regex парсит транзакции → Preview
3. Пользователь доволен → Импорт → Сохранить ParserConfig локально
4. Конфиг отправляется в Firestore (status: "unverified")
5. Верификация → попадает в Remote Config → все пользователи получают

## Статусы конфигов

| Статус      | Где хранится        | Кто видит                              |
|-------------|---------------------|----------------------------------------|
| generated   | Локально (DataStore)| Только этот пользователь               |
| unverified  | Firestore           | Пользователи этого банка (с пометкой)  |
| verified    | Remote Config       | Все                                    |

## Scoring конфигов (выбор лучшего из пула)

score = (parsed_count / expected_count)    // полнота
      × (valid_amount_ratio)               // корректность: суммы > 0, даты валидные
      × (hash_uniqueness_ratio)            // нет дубликатов хешей
      × user_confirmations                 // сколько пользователей подтвердили

Если confidence < 0.7 → показываем с предупреждением.

## Верификация

На старте — ручная (ты в Firebase Console / admin-панель).
Позже — auto-verify: 3+ разных пользователя успешно импортировали → verified.

Процесс:
- Пользователь импортировал → конфиг + sample (первые 5 транзакций, без PII) → Firestore
- Cloud Function → уведомление (Telegram/email)
- Проверка в admin-панели: regex, sample, результат
- Одобрение → Cloud Function пушит в Remote Config

## Риски и решения

| Риск                          | Решение                                              |
|-------------------------------|------------------------------------------------------|
| Плохой regex от Gemini        | Валидация: прогнать на sample, проверить parsed_count |
| ReDoS (зависание)             | Timeout 5 сек, regex complexity check                |
| Подмена конфига               | Ручная верификация на старте                         |
| Разные форматы одного банка   | bank_id + версия, несколько конфигов на банк         |
| PII в sample данных           | Не отправлять детали, только структуру               |

## Этапы реализации

### MVP (Phase 1)
1. Gemini генерирует ParserConfig по тексту выписки (промпт с примером ParserConfig)
2. Regex парсит → Preview → пользователь подтверждает
3. Сохранение конфига локально (Room или DataStore)
4. При следующем импорте этого банка — regex сразу, без Gemini
5. Ручное добавление в Remote Config тобой

### V2 (Phase 2)
1. Автоматическая отправка конфига в Firestore
2. Admin-панель для верификации
3. Cloud Function для пуша verified конфигов в Remote Config

### V3 (Phase 3)
1. Auto-verify по количеству подтверждений (3+ пользователей)
2. A/B testing конфигов (несколько regex для одного банка → выбор лучшего)
3. Версионирование конфигов (банк обновил формат выписки)

## Преимущества над прямым Gemini-парсингом

- Точность: regex детерминистичен, LLM — нет
- Дедупликация: стабильные хеши (одинаковый вход → одинаковый хеш)
- Стоимость: Gemini 1 раз на банк, а не на каждый импорт
- Оффлайн: после первого раза работает без интернета
- Масштабируемость: поддержка любого банка без ручного кода
